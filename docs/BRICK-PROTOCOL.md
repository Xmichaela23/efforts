# Brick Protocol — Triathlon

How the plan engine prescribes brick sessions (bike → run transition workouts) for triathlon athletes. Companion to `docs/SESSION-FREQUENCY-DEFAULTS.md` (per-discipline counts), `docs/SCHEDULING-RULES.md` (placement), and `docs/STRENGTH-PROTOCOL.md` (strength). Code: `src/lib/session-frequency-defaults.ts` (cap matrix + helper), `supabase/functions/generate-combined-plan/session-factory.ts:1129-1173` (factory), `supabase/functions/generate-combined-plan/week-builder.ts` (placement).

This doc captures the phase × hours-tier cap matrix, the placement rules, the Taper-W1 race-rehearsal exception, and the research provenance behind each cell.

---

## 1. Architecture

**Brick is an EXPLICIT, phase-gated session type.** It is generated by `session-factory.ts:brick()` as a *pair* of sessions — bike + run — both carrying the `'brick'` tag plus their sport tag (`['brick', 'bike', phase]` and `['brick', 'run', phase]`). Bricks are not retroactive same-day pairings of independently-placed bike + run sessions; they are intentional protocol artifacts.

Phase-gating is two-layered:
- **Cap** — `bricks_per_week_by_phase[phase]` in `SessionFrequencyDefaults` (tier-aware, populated by `computeSessionFrequencyDefaults` at the matrix layer).
- **Per-week lookup** — `brickCapForPhaseWeek(defaults, phase, weekInPhase)` in `src/lib/session-frequency-defaults.ts` is the canonical access point. Handles the Taper-W1 race-rehearsal exception and the never-emit-bricks-in-recovery/rebuild guard.

Week-builder consumers read the cap via the helper. Phase logic upstream (recovery weeks, race week, etc.) may set 0 unilaterally; the helper is a *cap*, not a floor.

## 2. Phase × hours-tier cap matrix

Values per `src/lib/session-frequency-defaults.ts:TRIATHLON_TIER_EXTRAS`. Numbers are the **count of brick session pairs per week** at that (phase, hours-tier) cell.

| Phase | 5-7 hr | 8-10 hr | 10-12 hr | 12-14 hr | 14+ hr |
|---|---|---|---|---|---|
| Base | 0 | 0 | 0 | 0 | 0 |
| Build | **1** | 1 | 1 | 1 | **1** |
| Race-specific | **1** | **1** | **1** | **2** | 2 |
| Taper W1 | 1¹ | 1¹ | 1¹ | 1¹ | 1¹ |
| Taper W2+ | 0 | 0 | 0 | 0 | 0 |
| Rebuild | 0 | 0 | 0 | 0 | 0 |
| Recovery | 0 | 0 | 0 | 0 | 0 |

¹ Taper-W1 brick is the race-rehearsal session; not from the matrix, applied via `brickCapForPhaseWeek` regardless of hours tier.

**Cell decisions (2026-05-11):**
- **Base = 0 all tiers** — matches Mottiv majority and avoids overcrowding low-volume weeks. Triathlete's 2 base-phase bricks (W2, W6) treated as outliers.
- **Build 5-7 hr = 1** — adopt user spec. All three reference plans prescribe 1 brick in build even at low volume; pre-fix the engine emitted 0, leaving 5-7-hr athletes without brick adaptation entirely.
- **Build 14+ hr = 1 (FIX)** — pre-fix the engine emitted 0 at 14+/build, a clear anomaly given all three references prescribe 1.
- **Race ≤12 hr = 1, Race >12 hr = 2** — tier-aware split. Lower tiers (5-7, 8-10, 10-12) match Mottiv + MyProCoach consensus; higher tiers (12-14, 14+) match Triathlete + MyProCoach Advanced peak.
- **Taper W1 = 1 race rehearsal** — matches Triathlete W19 brick. Mottiv and MyProCoach taper at 0; we follow Triathlete here because the race-rehearsal brick is widely-used coaching practice and verified by Friel's Triathlete's Training Bible.

## 3. Placement rules

### 3.1 Saturday-attached to long_ride

The brick run runs OFF the long ride. Implementation: when the cap is ≥1 for the week, the brick is placed on `long_ride_day` (typically Saturday). The long_ride session and the brick run share a calendar day; the run carries the `'brick'` tag so the same-day matrix (`schedule-session-constraints.ts:ROWS`) treats it as a brick pair rather than two independent sessions.

When the athlete's long_ride anchor is mid-week (e.g., Wednesday long_ride), the brick still attaches to the long ride — the day-of-week is athlete-driven, the *attachment* is the invariant.

### 3.2 Brick run duration by phase

The brick run length scales with phase (per `session-factory.ts:1138-1139`):
- **Build:** ~15–20 min easy off the bike — transition rehearsal at low intensity.
- **Race-specific:** ~30–50 min moderate — race-pace simulation.
- **Taper W1:** ~15–20 min easy — minimal volume, race-day rehearsal only.

Bike portion of the brick is the athlete's standard long_ride (no special truncation; the brick attaches to the existing long ride rather than replacing it).

### 3.3 Race-week brick (always 0)

Race week itself emits 0 bricks (`week-builder.ts:effectiveBricks = raceThisWeek ? 0 : bricksThisWeek`). The race is the brick. This is unconditional regardless of phase classification (a race week can still be classified `race_specific` for other purposes; the brick cap is overridden to 0).

### 3.4 Rebuild + Recovery weeks (always 0)

`brickCapForPhaseWeek` explicitly returns 0 for `phase === 'rebuild'` or `phase === 'recovery'`, regardless of how the matrix is populated. Post-race rebuild blocks need clean recovery; mid-cycle recovery weeks ("3:1" / "2:1" deload patterns) should not include the highest-fatigue session of the week.

## 4. Taper W1 race rehearsal (special case)

Pre-2026-05-11 the brick cap was phase-only — taper emitted 0 bricks uniformly. The user-locked spec adds a **W1 of taper = 1 brick** race-rehearsal exception.

Implementation: `brickCapForPhaseWeek(defaults, 'taper', weekInPhase)` returns `1` when `weekInPhase === 1`, else `0`. The cap matrix's `taper: 0` value is unchanged (it's the W2+ value); the W1 exception lives in the helper.

Caller responsibility: `week-builder.ts` computes `weekInPhaseForBrick` via the existing `weekInPhaseForTimeline` helper before consulting the cap. When `phaseBlocks` are not provided (legacy callers), `weekInPhase` defaults to 1 — meaning a legacy single-week taper still emits the rehearsal brick by default.

## 5. Research provenance (per-cell)

Cells sourced from empirical synthesis of three 70.3 reference plans (2026-05-11 fetch):

| Source | Plan | Sessions/wk | Brick prescription |
|---|---|---|---|
| Triathlete.com | [20-week 70.3 plan](https://www.triathlete.com/training/20-week-training-plan-first-70-3-triathlon/) | 5-6, 5 days/wk | Base W2,W6 / Build W9,W11,W13 / Peak W15,W17,W18 / Taper W19 |
| Mottiv | [18-week 70.3 plan](https://www.mymottiv.com/half-ironman-training-plan/18-week-half-ironman-training-plan) | 5-7 days/wk | Brick starts W7 (build), continues through peak; taper at 0 |
| MyProCoach | [Intermediate 12-week](https://www.myprocoach.net/free-training-plans/half-ironman-70-3/) | 6 days/wk | Weekly Saturday brick (all levels: Beginner, Intermediate, Advanced) |
| ROUVY | [70.3 beginner guide](https://rouvy.com/blog/half-ironman-training-plan) | 5-6 days/wk | Brick weekly in build/peak; race-week brick before taper |
| Active.com | 70.3 plan compendium | varies | Saturday brick standard across plans |
| Friel | *The Triathlete's Training Bible* 5th ed | varies | "Long run not the day after a long ride; brick is the weekend joint workout" — supports Saturday brick placement |

**Disagreement resolution** — Mottiv-style "no base bricks" preferred over Triathlete's W2/W6 base bricks because:
1. Two of three references (Mottiv + MyProCoach pre-build) agree on 0 base bricks.
2. Low-volume athletes (5-7 hr/wk) need pillar volume in base, not transition skill.
3. Brick adaptation can be deferred to build without compromising race readiness.

**Disagreement resolution** — MyProCoach's "weekly brick all year" not adopted because:
1. It conflates base-phase aerobic build with race-specific transition work.
2. The other two references treat base bricks as exceptional.
3. Adopting weekly bricks would inflate weekly TSS at low volumes, crowding out swim/run pillars.

## 6. Rationale for asymmetric phase prescription

Brick prescription scales with race-specificity, not generic training stress:

- **Base (0 bricks)** — adaptation focus. Aerobic capacity in each pillar built independently. Bricks would dilute the per-pillar specificity needed at base. Athletes new to the modality need clean swim/bike/run sessions to build technique; brick adds cognitive load that doesn't serve base goals.

- **Build (1 brick all tiers)** — transition fitness introduction. The athlete now has aerobic base; brick introduces the cycling-into-running transition that's unique to triathlon. One brick/wk gives ~12-16 brick exposures across a 4-month build/race cycle, sufficient for transition adaptation per *Friel 2016* + *Vleck et al 2014* (transitions are skill-acquired, not endurance-trained).

- **Race-specific (1 brick at low tiers, 2 at high tiers)** — race rehearsal. Two bricks per week at 12+ hr/wk allows both a "rehearsal" brick (race-day pacing) and a "neural" brick (faster transition). Below 12 hr/wk the schedule can't accommodate two without compromising long_ride or long_run volume; one brick is the consensus minimum.

- **Taper W1 (1 brick rehearsal)** — race-day simulation. The athlete needs a final transition rehearsal at race-pace effort. *Friel 2016* explicitly prescribes a "race-rehearsal brick" 10-14 days out from race day. This is W1 of a 2-week taper.

- **Taper W2+ (0 bricks)** — volume reduction. The athlete is sharpening; brick stress is counterproductive. Match Mottiv + MyProCoach standard taper practice.

- **Rebuild (0 bricks)** — recovery focus. Post-race rebuild needs aerobic restoration, not high-fatigue compounds. Brick would re-introduce the highest-load session of the week at the worst time. Code-level guard in `brickCapForPhaseWeek` prevents matrix tweaks from breaking this.

- **Recovery deload weeks (0 bricks)** — same logic. "3:1" loading pattern needs the recovery week to be genuinely lighter; brick on a recovery week makes it the hardest weekend.

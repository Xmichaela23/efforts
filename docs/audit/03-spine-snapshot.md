# AREA — Spine / state-trend / snapshot

## What this area does (plain-language overview)

This area is the app's "deterministic fitness brain." It turns raw per-workout facts into (a) a weekly aggregate row per athlete (`athlete_snapshot`, keyed `user_id + week_start`) and (b) a per-discipline trend verdict object (`state_trends_v1`) cached on the current week's snapshot row. The intent (SPEC-athlete-state-spine) is one source of athlete truth: code computes verdicts (improving / holding / sliding / needs_data) per discipline, plus load/ACWR/readiness/interference; every screen and the LLM narrative *describe* those verdicts rather than each re-deriving fitness. The verdict logic lives in pure modules under `_shared/state-trend/` and is run identically by the client STATE screen (`useStateTrends`) and the server (`compute-snapshot`), so the live screen and the cached spine cannot drift.

There are THREE distinct things named "snapshot" in this codebase; only the first is the spine (see Discrepancies):
1. `athlete_snapshot` TABLE — weekly aggregate, written by `compute-snapshot/index.ts`. THE SPINE.
2. `_shared/athlete-snapshot.ts` — a plan-generation `AthleteSnapshotV1` pinned into `plan.config` (frozen 1RMs/FTP/paces). Unrelated to the spine.
3. `_shared/athlete-snapshot/` directory — the LLM week-coaching "AthleteSnapshot" object (daily ledger + identity + body-response + coaching). A separate per-call construct, not the table.

## Features / flows

### Weekly snapshot aggregation (`compute-snapshot`)
- **What it does:** Aggregates 1 target week of `workout_facts` + `exercise_log` into one `athlete_snapshot` row, with trends vs the prior 4 weeks (chronic load, ACWR, RPE/pace/strength-volume trends, interference, intensity distribution, top lifts).
- **How it works:** `compute-snapshot/index.ts`. Fetches 5 weeks of facts (target + 28 prior days) and exercise_log for the same range (`index.ts:300-334`). Splits target-week vs prior-4-weeks; buckets prior facts into 4 individual weeks by Monday (`:343-352`). `aggregateWeek()` (`:86-228`) sums workload, RPE, readiness, per-discipline run/ride/strength signals, HR-zone seconds → intensity distribution. Chronic load = avg of the 4 prior weekly workloads; ACWR = current/chronic (`:362-368`). Trends (`pctChange`) computed vs the 4-week chronic for RPE, run easy-pace-at-HR, strength volume (`:376-399`). `buildTopLifts` (`:230-264`) finds best e1RM this week for the big-four lifts vs older best. Interference detection (`:407-482`) derives aerobicDirection + structuralDirection and flags dominance. Plan context + planned-session adherence (`:485-525`). Builds the snapshot object and UPSERTs on `(user_id, week_start)` (`:651-697`). A SEPARATE guarded UPDATE writes cycling CTL/ATL/TSB from the newest ride's `workout_analysis.fitness_v1` (`:706-744`).
- **Inputs / outputs:** Reads `workout_facts`, `exercise_log`, `readiness_checkins` (D-140/141), `planned_workouts`, and (for state_trends_v1) `workouts`, `route_progress_metrics`. Writes one `athlete_snapshot` row.
- **Triggers:** Fire-and-forget HTTP from `compute-facts/index.ts:1783` after every per-workout fact computation; `post-import-athlete-pipeline.ts:50` after Strava/Garmin import (milestone-gated per CLAUDE.md); `recompute-workout/index.ts:137` (D-178, on manual recompute, with the workout's own week); `generate-training-context/index.ts:645` lazily when the current-week snapshot is missing. Direct invoke accepts `{ user_id, week_start? }`; `week_start` defaults to `mondayOfToday()`.

### state_trends_v1 assembly + cache (`_shared/state-trend/` + compute-snapshot Step 4a)
- **What it does:** Produces one per-discipline performance verdict object (strength/run/swim/bike, bike carrying dual power+efficiency reads) and caches it on the CURRENT week's snapshot row.
- **How it works:** `assembleStateTrends()` (`assemble.ts:93-158`) is the ONE assembly. compute-snapshot Step 4a (`index.ts:583-649`) fetches the same rows the client hook fetches, with the same `STATE_TREND_WINDOWS`, flattens them identically, calls `assembleStateTrends`, then `toStateTrendsV1()` (`assemble.ts:197-215`) shapes it into the cache contract. Per-discipline computes: strength (`strength.ts`, per-lift e1RM → primary-lift roll-up), bike (`bike-fitness.ts`, terrain-binned 20-min power LEADS + HR-at-power efficiency secondary), run (`run.ts`, GAP pace at easy effort), swim (`swim.ts`, pace/100). Every discipline feeds the shared `classifyTrend` primitive (`classify.ts`). Thresholds resolved per athlete cadence by `resolveThresholds` (`thresholds.ts`).
- **Inputs / outputs:** Reads `exercise_log` (12wk), `workouts.workout_analysis.bike_fitness_v1` (latest 30 rides), `route_progress_metrics` joined with `workouts.workout_analysis.classified_type` (6wk), `workout_facts.swim_facts` (8wk), `planned_workouts`/`workouts` (7d adherence, 90d cadence). Writes `athlete_snapshot.state_trends_v1` (JSONB).
- **Triggers:** Only inside compute-snapshot, and only when `targetWeek === mondayOfToday()` (`index.ts:584`). Historical weeks leave it null.

### The shared trend primitive (`classify.ts`)
- **What it does:** Classifies any dated metric series as improving/holding/sliding/needs_data with noise + staleness guards.
- **How it works:** `classifyTrend()` (`classify.ts:40-99`). Filters to value>0, in-window, post-exclude, sorted. Three-layer noise guard: min-session gate → needs_data; endpoint smoothing (avg of 2 earliest vs 2 most-recent); dead-band between slidePct/improvePct → holding. `lowerIsBetter` flips the sign for verdict only (pctChange stays raw). Staleness gate: a real verdict whose newest point is older than `freshnessDays` decays to needs_data with `stale:true`.
- **Inputs / outputs:** Pure function — `(points, thresholds, asOf, opts)` → `TrendResult`. No Date.now / fetching.
- **Triggers:** Called by every discipline adapter (strength/bike/run/swim) on both client and server.

### Fitness-direction roll-up + headline (`rollupFitnessDirection`, `headline.ts`, `load-headline.ts`)
- **What it does:** Collapses the four per-discipline verdicts to one `FitnessDirection` (improving/stable/declining/mixed) for the coach, and synthesizes a two-part STATE headline.
- **How it works:** `rollupFitnessDirection()` (`assemble.ts:182-193`): ignores needs_data; improving+sliding → mixed; else the single direction; else stable. `synthesizeHeadline()` (`headline.ts:33-77`) builds "Building — strength up, run sliding" from performance-leading cards, gating swim out (`HEADLINE_GATED_DISCIPLINES`). Client `load-headline.ts` builds the glance line from ACWR label + readiness state + fitness direction (slots STATE · FITNESS — OBSERVATION).
- **Inputs / outputs:** Reads cached `state_trends_v1`; outputs strings/enums.
- **Triggers:** coach (`index.ts:2705`), session-detail/build pass-through, client STATE/home.

### Client consumers
- `useStateTrends.ts` — fetches rows client-side and calls `assembleStateTrends` for the LIVE STATE screen (not the cache).
- `useAthleteSnapshot.ts` — reads `athlete_snapshot` rows directly (last N weeks) for load/trend display.
- `fetch-arc-context.ts` / `build-arc-line.ts` / `load-headline.ts` — Arc/home line + glance headline (arc-context is the server reader of the snapshot table).

## Edge cases & conditional handling

- **state_trends_v1 only on current week** — `compute-snapshot/index.ts:584`: `if (targetWeek === mondayOfToday())`. Any invocation with an explicit past `week_start` (e.g. `recompute-workout` passing the workout's own week, `index.ts:137`) writes the row WITHOUT state_trends_v1 (null). Recomputing an old week never refreshes the current-week spine.
- **state_trends_v1 failure is non-fatal** — wrapped in try/catch (`index.ts:585-648`); on error logs and sets `stateTrendsV1 = null`, snapshot still upserts.
- **CTL/ATL/TSB write is separate + non-fatal** — `index.ts:706-744`: written via a guarded UPDATE (not the main upsert) so an unmigrated column can't break the snapshot. Skipped silently if no ride carries `fitness_v1`. TSB falls back to `ctl − atl` when `tsb` absent (`:727`).
- **Readiness rollup fallback (D-140/141)** — `index.ts:544-572`: prefers `readiness_checkins` table over the target week; falls back to facts-derived `current.avgReadiness` when the table is absent/errors/empty. Whole block try/catch'd.
- **Cold start / no facts** — `aggregateWeek([])` returns zeros/nulls; ACWR null when chronicLoad null or 0 (`:366`); pctChange null when baseline null/0 (`pctChange`, `:49-52`). `rollupFitnessDirection(null)` → 'stable' (matches prior catch-all default, `assemble.ts:182-186`).
- **classify min-session gate** — below `minSessions` → needs_data, never a guess (`classify.ts:65-67`). `minSessions` scales 3–5 by available sessions in window (`thresholds.ts:50-51`).
- **classify staleness gate** — newest qualifying point older than `freshnessDays` → needs_data + `stale:true` (`classify.ts:89-96`). `freshnessDays` cadence-scaled, clamped 7–35 (`thresholds.ts:49`); omitting it disables the gate.
- **Endpoint overlap at n=3** — `k = min(2, n)`, so early/recent pairs share the middle point at n=3 → conservative pctChange (documented benign in SPEC-athlete-state-spine; surfaced as `provisional`).
- **Deload exclusion** — `isDeloadWeek` (`deload.ts:15-18`) drops points whose `meta.name` matches `/deload/i` from strength/run/swim trends (NOT bike power/efficiency — those classify calls pass no `exclude`).
- **Run intent gate / corrupt-pace guard** — only `classified_type === 'easy'` runs are comparable (`run.ts:27`); the gate reads `workout_analysis.classified_type` (joined), NOT the null `route_progress_metrics.workout_intent`. Plausibility band 150–750 s/km drops corrupt GAP values (`run.ts:35-36,50`) — comment notes one 2280 s/km row that flipped the trend.
- **Swim Q-038 guard** — pace band 40–240 s/100m drops implausible points (`swim.ts:23-24`); swim is also gated out of the headline lead (`headline.ts:31`). In practice swim mostly → needs_data → adherence axis.
- **Bike power binning** — only `climbing` and `flat_sustained{threshold,sweet_spot,tempo}` bins trend; vo2/endurance excluded (`bike-fitness.ts:20-23`). Surfaces the FRESHEST bin with a real verdict (`:68-82`). Power leads; falls back to efficiency verdict when power = needs_data (`assemble.ts:108-110`).
- **Provisional flags** — `isProvisionalTrend` (`bike-fitness.ts:41-50`): sampleCount 3–4 OR clustered <21d span → provisional. Strength provisional when <2 primaries or any primary provisional (`assemble.ts:136-138`).
- **Adherence label is count-only** — never a judgment word ("0/2 planned", "1 unplanned", "none this week"), to avoid mimicking a trend verdict (`adherence.ts:43-47`).
- **DISPLAY_MODE fallback** — `discipline.ts:38`: adherence shown only where performance is absent today; `'co-equal'` is a one-line flip awaiting Layer-1 context tags (`SessionContextTag` is an empty placeholder, `adherence.ts:6-16`).
- **Interference fallback ladder** — `compute-snapshot/index.ts:423-459`: aerobicDirection prefers run-easy-pace trend, then run-efficiency delta, then run-workload delta; structuralDirection prefers avg-1RM delta, then strength-volume trend. interference null unless BOTH directions resolve (`:463`).
- **Plan week number best-effort** — only set if a planned_workout_id resolves a `week_number` (`:497-506`); `plan_phase` is always null in the upsert (`:489`,`:685`) — never populated.
- **bike row efficiency basis** — `assemble.ts:107`: `bikeFitness.efficiency.basis` set to the first non-null `band_source` across bike rows, else null.

## Redundancies / duplication (observed, not judged)

- **Two bike fitness paths.** `bike-fitness.ts` (`computeBikeFitness`, terrain-binned + HR@power) feeds the SPINE. `bike.ts` (`computeBikeState`, `pickBestPwr20`, `pwr20ToSeries`) is the older single-type `pwr20_trend_v1` adapter still consumed by `analyze-cycling-workout/index.ts:2502` and referenced in `cycling-v1/ai-summary.ts:196`. Both are exported from `state-trend/index.ts`. BUILD-SEQUENCE Phase-2a notes bike-fitness "supersedes the shipped single-type pickBestPwr20" — both still ship.
- **Client/server fetch+flatten duplicated by design.** `useStateTrends.ts:45-133` and `compute-snapshot/index.ts:589-643` independently fetch and flatten the same rows for the same `assembleStateTrends`. This duplication is the explicit "identical rows → identical output" single-source proof, but the two fetch blocks are maintained separately (e.g. the run classified_type join exists in both).
- **Ride EF chronic computed but unused.** `compute-snapshot/index.ts:398-399` computes `priorRideEFs` / `chronicRideEF`; no `ride_efficiency_factor_trend` is derived or written (contrast `run_easy_pace_at_hr_trend`, `strength_volume_trend`, `rpe_trend`).
- **pctChange computed twice in buildTopLifts.** `compute-snapshot/index.ts:257-258` calls `pctChange(...)` up to three times for the same lift's trend string.
- **Two adherence computations.** Weekly plan adherence is computed in compute-snapshot (`adherence_pct`, `:523-525`, sessionCount/planned) AND per-discipline in `adherence.ts` (`computeAdherenceState`) for state_trends — different windows (full target week vs trailing 7d) and granularity.
- **Three "snapshot" namespaces.** See overview — `athlete_snapshot` table vs `_shared/athlete-snapshot.ts` (plan-pinned) vs `_shared/athlete-snapshot/` (LLM coaching object).
- **disciplineOf duplicated.** `assemble.ts:38-45` (`disciplineOf`) and the client hook both classify discipline strings; the snapshot's own `aggregateWeek` separately branches on `f.discipline === 'run'|'ride'|'bike'|'strength'` inline.

## Discrepancies & flags (for human review)

- **state_trends_v1 staleness via current-week gate.** Because the cache is only written when `week_start === mondayOfToday()`, a recompute or import that targets a PAST week leaves the live current-week `state_trends_v1` untouched. `recompute-workout` deliberately passes the workout's own week (`recompute-workout/index.ts:137-139`), so recomputing an OLD workout will NOT refresh the spine verdict even though its comment claims recompute "reaches every cached reader … the athlete-state spine." Worth confirming the spine refresh actually fires for old-workout recomputes. (Flag — trigger/intent mismatch, not verified.)
- **`plan_phase` never populated.** Declared and upserted as null (`:489`,`:685`); `plan_week_number` only set opportunistically. Consumers reading `plan_phase` from the snapshot would always get null.
- **`useAthleteSnapshot.ts` type omits many written columns.** The `AthleteSnapshotRow` interface (`:4-36`) lacks `state_trends_v1`, `ride_long_ride_duration`, `ride_interval_adherence`, `interference`, `intensity_distribution`, `ctl`/`atl`/`tsb`. The query is `select('*')` so the data is present at runtime; the typed surface just doesn't expose them. Possibly intentional (this hook is the load/trend display, not the spine reader) — flagged as a typed/written mismatch.
- **Manual SQL migrations / migration-tracking divergence.** Comments at `index.ts:704-705` and `:741-743` state CTL/ATL/TSB columns were applied "manually via SQL editor," and the readiness rollup (`:537-543`) is written to deploy safely before its migration lands. Schema state can't be verified from code alone — flagged that migrations may be out of band.
- **Run trend reads a known-corrupt upstream.** `run.ts` comments document that `route_progress_metrics.workout_intent` is null at source (compute-facts:930 reads an unpopulated field) and that GAP values can be corrupt upstream (compute-facts GAP computation). The spine guards defensively but the root causes are flagged as tracked-separately. (Cross-area: see analyzers / compute-facts audit.)
- **`generate-daily-context` is empty.** The directory `supabase/functions/generate-daily-context/` contains no `index.ts`. Listed in the work order's concrete-code set; reachable/intended status unclear from code.
- **Context functions don't read the spine.** `generate-overall-context`, `generate-training-context` do NOT read `state_trends_v1` (grep-confirmed); `generate-training-context` only LAZILY triggers compute-snapshot when the current-week row is missing. The "coach reads the spine verdict" wiring (SPEC build order step 2/3) lands in `coach/`, `session-detail`, `workout-detail`, `narrative-core/adapters/coach.ts` — not these context generators. Confirm whether the context generators were meant to be on the spine.
- **bike efficiency `improvePct/slidePct` overridden to ±3.** `bike-fitness.ts:36-38` builds `efficiencyThresholds` by spreading `resolveThresholds('bike', spw)` then overwriting improve/slide to ±3 and `lowerIsBetter:true`. The universal bike thresholds (±2, `thresholds.ts:15`) are silently replaced for the efficiency signal only — matches SPEC table but is an inline override worth noting.
- **`isProvisionalTrend` returns false for needs_data but the cache stores provisional from perfByDisc only.** `toStateTrendsV1` reads `provisionalByDisc` (`assemble.ts:200`), which for bike uses power-or-efficiency provisional; the per-discipline `provisional` in the cache may not match the leading signal in every branch. Worth a spot-check against the live STATE screen.
- **No test on `assembleStateTrends` / the spine cache.** `compute-snapshot/index.test.ts` covers `aggregateWeek` (interval adherence, long-run/ride duration) only. There is no `state-trend/*.test.ts` (grep-confirmed none) exercising `assembleStateTrends`, `classifyTrend`, or `toStateTrendsV1`, despite the single-source claim resting on client/server parity. (Flag — coverage gap for the load-bearing parity guarantee.)
